ScholarPath — رحلة الموديل

من الـ ERD الأساسي → EERD (تحسين) → Relational Mapping → Class diagram. المثال الجاري هو الكيان USER.

ERD — النموذج المفاهيمي الأساسي. الـUSER كيان واحد مسطّح، والدور (Role) مجرّد attribute. مفيش تخصّص ولا كيانات ضعيفة.

بنرسم العالم كـكيانات (مستطيلات) + علاقات (معيّنات) + attributes (بيضاوي، والمفتاح مُسطّر).
  • هنا USER كيان واحد، والأنواع المختلفة من المستخدمين مش متمايزة.
  • كافي للفهم العام، لكنه مش بيعبّر إن لكل دور بيانات مختلفة → ده اللي بنحلّه في الخطوة الجاية.

EERD — التحسين: الـUSER اتخصّص لأربع أدوار عبر ISA. الدائرة o = overlapping، والمشاركة partial.

ده قلب الموضوع — بنطبّق أدوات الـEER:
  • Specialization / ISA: USER بيتقسّم لـ Student / Company / Consultant / Admin.
  • الدائرة o = overlapping: المستخدم ممكن يحمل أكتر من دور (Student يترقّى لـConsultant ويفضل الاتنين).
  • Partial: المستخدم الجديد (Unassigned) لسه مش أي نوع.
  • وبنضيف كمان weak entities (سجلات التعليم/الـchildren)، multivalued (الدول المفضّلة JSON)، وderived (التقييم المتوسط، عدّاد التصويت).

Mapping — تحويل التخصّص لجداول بخوارزمية Elmasri. Option C (single-table).

التخصّص لازم يبقى جداول علائقية. اخترنا Option C (single-table):
  • جدول Users واحد للأب.
  • جدول وصل UserRoles (M:N) للأدوار.
  • جدول UserProfiles واحد بيلمّ أعمدة كل الأدوار — واللي مش بينطبق على دورك يبقى NULL (بدل جدول لكل نوع).

Class — نفس الموديل بالـOOP. ApplicationUser بيورث IdentityUser<Guid> (مش BaseEntity).

أخيرًا نعبّر عنه ككلاسات C#:
  • ApplicationUser بيورث IdentityUser<Guid> (مش BaseEntity — نقطة مهمة).
  • الـUserProfile (علاقة 1 : 0..1) بيحمل حقول الأدوار الـnullable.
  • وده بيركب فوق الـClean Architecture (ports & adapters): الـApplication بيعرّف الواجهات، والـInfrastructure بيقدّم التطبيقات.
