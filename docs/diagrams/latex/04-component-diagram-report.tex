\begin{abstract}\noindent
This report describes the runtime structure of ScholarPath as a UML component
diagram in the style of Sommerville. ScholarPath is a Clean Architecture
monolith (\texttt{API} $\rightarrow$ \texttt{Application} $\rightarrow$
\texttt{Domain}; \texttt{Infrastructure} $\rightarrow$ \texttt{Application}/\texttt{Domain})
with a separate React single-page-application client and a set of external
Azure / third-party services.
\end{abstract}

\tableofcontents\bigskip

\section{Notation}
\begin{itemize}
  \item A \textbf{component} is a box (here, a deployable assembly or external
        service). Components are grouped by their \emph{deployment node}
        (Azure Static Web Apps, Azure App Service, external services).
  \item A \textbf{provided interface} is drawn as a lollipop ($\circ\!\!-$);
        a \textbf{required interface} as a socket ($-\!\!\subset$). In the source
        these are the application \emph{ports} and the infrastructure
        \emph{adapters} that provide them.
  \item A \textbf{dependency} (arrow) means one component uses another.
        Clean-Architecture dependencies point \emph{inward}.
\end{itemize}

\section{Notable data flows}
\begin{itemize}
  \item \textbf{Auth + API:} the React SPA calls the API over HTTPS with a JWT,
        and subscribes to the SignalR hubs over WebSocket for realtime updates.
  \item \textbf{Payments:} the Infrastructure layer creates / captures / refunds
        Stripe PaymentIntents; Stripe calls \emph{back} into the API via a webhook,
        which is verified and recorded idempotently.
  \item \textbf{Analytics (reverse-ETL):} domain events are published to Azure
        Event Hub and consumed by Power BI; a daily Power BI dataflow writes
        churn-risk scores back into the \texttt{UserRiskFlags} table in Azure SQL.
\end{itemize}
