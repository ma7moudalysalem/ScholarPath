# API audit — 2026-05-21

**Scope.** Every controller under
`server/src/ScholarPath.API/Controllers/` (23 files): authorization,
error handling, input validation, rate limiting, response shape, security.

## Verdict

**Minor issues, mostly fixed inline.**

The API surface is in good shape. The MediatR validation pipeline is wired
correctly, FluentValidation covers every command, the ExceptionHandlerMiddleware
maps the application-level exceptions to RFC 7807 ProblemDetails, and the
Admin / Consultant / Company / Student role gates are present on the
right endpoints. CORS uses a whitelist (no wildcards). No raw SQL, no
`.Result` / `.Wait()`, no password / JWT logging.

The issues found were of two kinds:

1. **One bug-with-teeth** — `UnauthorizedAccessException` thrown by 15
   handlers was leaking as HTTP 500 because the middleware did not catch
   it. Fixed by mapping it to 403.
2. **Defense-in-depth gaps** — three controllers had no class-level
   `[Authorize]` and relied on per-endpoint attributes; admin-only role
   strings missed `SuperAdmin` in a handful of places; AI / chat endpoints
   had no rate limiting. All addressed.

No big architectural changes were made and no new packages were added.

## What was verified clean

- **No SQL injection.** No `FromSqlRaw` / `SqlQuery` / `FromSql` anywhere
  in the controllers (`Categories` uses LINQ-to-EF, which parameterises).
- **No async-over-sync.** No `.Result` / `.Wait()` / `GetAwaiter().GetResult()`
  anywhere in the controllers.
- **CancellationToken propagation.** Every controller action accepts a
  `CancellationToken` and forwards it to MediatR; `ConfigureAwait(false)`
  is used inside the handlers.
- **CORS is whitelisted.** `Cors:AllowedOrigins` is read from
  `appsettings.json` (defaults to `localhost:5173` / `localhost:3000`),
  never `AllowAnyOrigin`. `AllowCredentials()` is on, so a wildcard
  would be a runtime error anyway.
- **No secrets logged.** A grep of `password`, `secret`, `token` across
  every controller turned up only parameter names — no logger calls
  capture passwords / JWTs / Stripe webhook secrets.
  `WebhooksController` logs `RemoteIpAddress` on a webhook parse
  failure, which is fine.
- **File uploads are validated.** `DocumentsController.Upload` enforces
  `[RequestSizeLimit(26 MB)]`, `UploadDocumentCommandHandler` checks the
  extension allowlist (`AllowedExtensions`), enforces the 25 MB cap, and
  runs the antivirus scan **before** writing to storage. The scan is
  fail-closed — an unscannable file is rejected.
- **FluentValidation coverage is good.** Every command/query that takes
  user input has a validator: chat, community, profile, payments,
  bookings, applications, AI, auth, financial config, notifications,
  scholarships. `MaximumLength` is applied to free-text fields (e.g.
  `SendMessage.Body` ≤ 2000, `CreatePost.BodyMarkdown` ≤ 10000,
  `AskChatbot.Message` ≤ 2000).
- **Error mapping is consistent.** The ExceptionHandlerMiddleware maps:
  `ValidationException → 422`, `BookingDomainException → 422`,
  `NotFoundException → 404`, `ConflictException → 409`,
  `ForbiddenAccessException → 403`,
  `ServiceUnavailableException → 503`, fallback `→ 500`. With this
  audit's fix, `UnauthorizedAccessException → 403` is also covered.
- **Paged endpoints return `PagedResult<T>`** consistently (admin lists,
  payments, audit log, notifications, scholarship moderation, community
  flagged, etc.). `PaginatedList<T>` is used in a couple of places
  (`ResourcesController.Search`, scholarship `Get`) — minor
  inconsistency, see follow-up below.
- **Security headers.** `SecurityHeadersMiddleware` sets
  `X-Content-Type-Options`, `X-Frame-Options=DENY`,
  `Referrer-Policy=strict-origin-when-cross-origin`,
  `Permissions-Policy`, `Cross-Origin-Opener-Policy`,
  `Cross-Origin-Resource-Policy`, and HSTS on HTTPS responses.
- **JWT validation.** RS256 with key from `IJwtKeyProvider` (Key Vault
  in prod, ephemeral in dev), `RoleClaimType = "active_role"` so role
  checks respect the dual-role model after a refresh.

## Recommendations for follow-up

These are not blockers — none change behaviour or fix a bug — but they
would reduce mental overhead in the future. They are intentionally
**not** done as part of this audit to avoid scope creep.

1. **Consolidate `PagedResult<T>` and `PaginatedList<T>`.** Two
   parallel pagination DTOs exist in
   `Application/Common/Models/PaginatedList.cs` and inside
   `Application/Common/Interfaces/IAdminReadService.cs`. They're shaped
   slightly differently and clients have to handle both. Pick one
   (`PagedResult` is more widely used) and migrate the half-dozen
   `PaginatedList` consumers.
2. **Centralise `Accept-Language` parsing.** `ScholarshipController`
   inlines the same `Request.Headers.AcceptLanguage…` parsing 4 times,
   and `ApplicationsController.GetMyApplications` does it once more.
   A small extension method on `HttpRequest` (or a strongly-typed
   `IAcceptLanguageResolver` service) would dedupe this. Low risk,
   purely a refactor.
3. **`ApplicationsController.UpdateExternalStatus`** mutates only the
   `Status` enum — consider a `MaximumLength` on the enum's underlying
   string serialisation in OpenAPI to surface the allowed values to the
   client SDK generator.
4. **Hangfire dashboard authorization filter** uses reflection to find
   the `HttpContext` (`Program.cs:388`). It works but is brittle — when
   Hangfire updates the dashboard context API this will silently start
   returning `false`. Worth tracking for a runtime test or upgrading
   to a typed accessor when the Hangfire team ships one.
5. **`MeetingRecordingWebhookController`** is anonymous (Event Grid is
   not a logged-in caller) and trusts the request. The endpoint matches
   only known ACS event types and validates the subscription handshake,
   but it does not verify the Event Grid signature. Stripe's webhook
   does verify HMAC. Adding signature verification (or restricting the
   route to a known IP range via App Service access restrictions) is
   the right long-term fix.
6. **`DiagnosticsController.PushTestNotification`** is `[Authorize]`
   only — anyone signed in can spam SignalR notifications to themselves.
   Harmless today but if a customer is logged in, a noisy
   pen-test could send a flood. Consider `[Authorize(Roles =
   "Admin,SuperAdmin")]` or attach it to the `"ai"` (or a new
   `"diagnostics"`) rate-limit policy.
